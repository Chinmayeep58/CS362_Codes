Distributed Bank Backend
Latency and Inter-Process Communication
/universityThe Dilemma
You and a friend are opening aJoint Fixed Depositaccount. The campus bank uses a
modern distributed backend to process all financial data.
For security and load balancing, the bank assigns the processing of joint accounts totwo
independent server branches (Node A and Node B).
•Node A:Processes Client 1’s portion of the funds.
•Node B:Processes Client 2’s portion of the funds.
In real-world market dynamics, the bank applies a daily randomized market fluctuation
to the interest rate. Furthermore, strict banking regulations require a mandatory "crypto-
graphic audit" for every single daily transaction. This introduces an intentional processing
delay, causing the servers to take a noticeable amount of time to finalize the account.
/codeThe Distributed Calculation
Write a concurrent program where two computational nodes process compound interest
independently before sharing their results to find the total.
The Setup:
•Client 1 Principal (P 1):Derived from the last 4 digits of your phone number.
•Client 2 Principal (P 2):Derivedfromthelast4digitsofyourpartner’sphonenumber.
•Timeframe:N years, processedday-by-day.
The Flow:
1.Phase 1 (Calculation & Delay):Each Node computes its portion day-by-day. As
part of regulatory audit, each day’s calculation must include a brief delay.
2.Phase 2 (Randomness):Each Node applies a unique, randomized micro-bonus to
their interest rate based on their assigned phone number seed, guaranteeing unique
outputs for everyone.
3.Phase 3 (Communication):The Nodes exchange their finalmaturity valuesusing
a message-passing technique.
4.Phase 4 (Consensus):Both Nodes add the two values together. If both Nodes arrive
at the exact same Final Total, the transaction is successful!

Lab Concept Explanation:
•Parallelism through Threads:When a client connects, the server spawns a new
threading.Thread. This allows the server to handle Client 2 without waiting for
Client 1 to disconnect.
•The Race Condition (Mutex Lock):If Client A and Client B both try to
withdraw$500 at the exact same millisecond, both threads might read the balance
as$1000, approve the transaction, and update the balance to$500, essentially losing
track of one transaction. Thewith lock:block ensures that onlyone thread at a
timecan read and modify thebalancevariable.

Lab Concept Explanation:
•SPMD Architecture:Unlike Program 1 where the Server and Client had different
code files, in MPI,all nodes run the exact same script. They know how to behave
based on theirrank(Node 0 acts as Master, Nodes 1, 2, 3 act as workers).
•Distributed Reduction (comm.reduce):Instead of Node 0 sequentially asking
Node 1, then Node 2, then Node 3 for their totals,reduceutilizes network topology
to sum up the variables across the distributed network in parallel. This is the
foundation of technologies like MapReduce (Hadoop/Spark).
